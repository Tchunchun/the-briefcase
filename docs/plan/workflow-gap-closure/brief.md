**Status: implementation-ready**
## Problem
The framework is partially optimized for a Notion-backed workflow, but several core planning and delivery steps are still incomplete or inconsistently modeled. Brief updates are lossy in the Notion backend, artifact links are not reliably captured, review outcomes are stored as notes instead of structured workflow state, and the lifecycle from new work to shipped work is not fully traceable.
## Goal
Make the framework operationally complete for a Notion-first project so every role can read and update the right artifacts, every workflow transition has durable state, and required planning and ship artifacts remain linked from idea creation through release.
## Acceptance Criteria
- [ ] Notion-backed briefs can be created and updated without losing status, non-functional requirements, open questions, or technical approach content.
- [ ] The CLI exposes durable identifiers and URLs needed for agents to attach brief links and parent-child relationships when creating or updating Idea, Feature, and Task rows.
- [ ] The workflow has an explicit, documented, and implementable lifecycle for Idea, Feature, Task, review verdict, and route state from initial capture through shipped work.
- [ ] Review outcomes are persisted in a structured way that delivery-manager and implementation can route from without relying only on free-form notes.
- [ ] Required PM and operational artifacts are documented and linkable, including brief, related backlog items, decisions, review outcome, and release notes.
- [ ] PLAYBOOK and role skills align with the actual CLI/backend behavior for a Notion-first project and no longer instruct agents to perform unsupported linking or state transitions.
## Out of Scope
Replacing Notion with a different cloud backend; redesigning the five-role framework itself; broad UI or presentation work beyond the artifacts needed for workflow completion; migrating every historical artifact unless required to support the new lifecycle model
## Open Questions
All resolved. Review verdict will be modeled separately from Feature execution status, release notes remain page-based in Notion, and the minimum backwards-compatible change set is to extend the current backlog/CLI model rather than introduce a new heavy workflow artifact system.
## Technical Approach
Treat this as a workflow-completion feature for PM and operational artifacts only. Notion remains the source of truth for briefs, backlog tracking, review/routing state, release notes, and user-facing operational docs. Local Git remains the source of truth for implementation assets and engineering-governance docs such as `_project/tech-stack.md`, `_project/testing-strategy.md`, `_project/definition-of-done.md`, source code, tests, and code-adjacent ADRs.
Implement in five ordered slices.
Slice 1: brief persistence. Keep the recently-fixed Notion brief body replacement path and complete the round-trip contract so `agent brief read` and `agent brief write` preserve Status, Problem, Goal, Acceptance Criteria, Out of Scope, Open Questions, and Technical Approach. Add regression tests for create, update, and re-read behavior. This is the prerequisite for the rest of the feature because briefs must be editable safely before they can act as the source of truth.
Slice 2: durable PM artifact linking. Extend brief CLI responses to include both durable Notion identifiers and URL fields. Standardize backlog linking around existing `Parent` relation plus explicit URL fields: keep `Brief Link`, add a release-note link field for shipped features, and ensure parent-child relationships are maintained for Idea -> Feature and Feature -> Task. The goal is to make link creation supported by the CLI rather than by manual URL hunting.
Slice 3: structured review and routing state. Add explicit backlog properties and CLI support for `Review Verdict` and `Route State`. `Review Verdict` should be a dedicated select field on Feature rows with values `pending`, `accepted`, and `changes-requested`. `Route State` should be a dedicated select field used by delivery-manager to record the latest routing outcome, with values such as `routed`, `returned`, and `blocked`. Review findings may still live in notes, but routing decisions must no longer depend on parsing notes for verdicts.
Slice 4: explicit lifecycle vocabulary. Keep workflow state on separate axes rather than collapsing everything into one overloaded status field. Feature execution status should track work progression; review verdict should track acceptance; route state should track handoff outcome; release-note linkage plus terminal feature status should mark shipment. The recommended execution vocabulary is `draft`, `architect-review`, `implementation-ready`, `in-progress`, `review-ready`, and `shipped` for Features, while Task statuses remain `to-do`, `in-progress`, `blocked`, and `done`. `accepted` should not be used as a Feature Status because it belongs to review verdict, not execution.
Slice 5: docs and skill alignment. Update PLAYBOOK and the role skills so they reflect the real artifact boundary and the new structured fields. Skills should state clearly that Notion owns PM/ops artifacts, local Git owns implementation and engineering-governance assets, and the supported CLI fields are the only write path for Notion-backed planning data.
For backwards compatibility, keep the current unified backlog model and extend it incrementally rather than replacing it. Optional new CLI fields should degrade safely on the local backend until local markdown rendering is extended. No new dependencies are required, and there is no meaningful incremental cost beyond the existing Notion API usage because the feature works within the current databases/pages model.
