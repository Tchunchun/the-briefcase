**Status: implementation-ready**
## Problem
All CLI commands require `python3 -m src.cli.main <command>`, which is fragile (assumes `src/` is on the path, assumes the right interpreter is active) and verbose. AGENTS.md and skill files document this path, but it breaks on consumer installs where `src/` lives inside `.briefcase/`. There is no idiomatic `./agent` entry point at project root.
## Goal
Typing `./agent <command>` at project root works identically to the current `python3 -m src.cli.main <command>` invocation. AGENTS.md, PLAYBOOK.md, and all skill files use `./agent` exclusively. The entry point is robust: it finds the correct Python interpreter and `.briefcase/` location without the user managing paths.
## Acceptance Criteria
- [ ] `./agent` file exists at project root and is executable (`chmod +x`)
- [ ] `./agent <command>` produces identical output to `python3 -m src.cli.main <command>` for all existing commands
- [ ] `./agent` resolves the correct Python interpreter: prefers `.briefcase/.venv` if present, falls back to system Python 3
- [ ] `./agent` resolves CLI module from `.briefcase/src/` (not a hardcoded `src/` at project root)
- [ ] AGENTS.md commands table updated: all entries use `./agent <command>`
- [ ] PLAYBOOK.md updated: all CLI invocations use `./agent <command>`
- [ ] All five skill `SKILL.md` files updated: all CLI invocations use `./agent <command>`
- [ ] Running `./agent` with no arguments prints help (same as current behaviour)
- [ ] Works from any subdirectory inside the consumer project by walking up to the `.briefcase/` sentinel, or fails with a clear error if run outside a project
- [ ] Unit tests or integration smoke test confirms `./agent inbox list` returns valid output against a configured backend
## Out of Scope
Changing the CLI command structure or flags; adding new CLI commands; creating the install script that places `./agent` (that is the install script brief); Windows `.bat` or PowerShell equivalent in this version
## Open Questions
All resolved. `./agent` will be generated by install as a bash wrapper rather than committed to the framework repo, it will locate the consumer project by walking up to the `.briefcase/` sentinel, and it will inject `.briefcase/` onto `sys.path` so imports continue to resolve from `.briefcase/src/` without changing the framework package layout.
## Technical Approach
Generate `./agent` at install time rather than committing it to the framework repo. The script should be a bash wrapper because interpreter resolution is environment-specific per consumer install and shell is the simplest v1 mechanism for selecting `.briefcase/.venv/bin/python` when present, then falling back to `python3`.
The wrapper should resolve the project root dynamically by walking up from the current working directory until it finds `.briefcase/`. That keeps invocation stable from subdirectories and makes `.briefcase/` the single runtime/config sentinel. If no `.briefcase/` directory is found, exit with a clear error that the command must be run inside an initialized consumer project.
Once the project root is found, the wrapper should execute Python with a small bootstrap that inserts `${PROJECT_ROOT}/.briefcase` at the front of `sys.path` and then invokes `src.cli.main`. This preserves the existing `from src...` import contract while making `.briefcase/src/` the effective runtime source tree. No framework import rewrites are required.
This feature depends on the `.briefcase/` namespace-isolation decisions: `.briefcase/` is a post-install artifact, `.briefcase/storage.yaml` is the runtime config location, and `.briefcase/` is the sentinel used for project discovery. No new dependencies or paid services are introduced.
