**Status: implementation-ready**
## Problem
Onboarding a consumer project requires manually copying multiple folders (`src/`, `skills/`, `template/`), creating an agent entry point, updating `.gitignore`, and configuring `storage.yaml`. There is no single command to do this. The manual process is error-prone and produces inconsistent installs: wrong paths, missing gitignore entries, and unclear `.env` token naming.
## Goal
A single install command onboards a consumer project correctly and repeatably: copies the full framework into `.briefcase/`, writes `./agent` at project root, appends deliberate `.gitignore` entries, and places `storage.yaml` inside `.briefcase/`. Running it a second time overwrites `.briefcase/` only, leaving all consumer files untouched.
## Acceptance Criteria
- [ ] install can be invoked as `curl <url> | sh` or `./install.sh`; Python module install form is out of scope for v1
- [ ] On first run: copies all framework code into `.briefcase/` (`src`, `skills`, `template`); creates `./agent` entry point at project root; appends `.briefcase/`, `.briefcase/storage.yaml`, and `.sync-manifest.json` to `.gitignore` with explanatory comments; creates `.briefcase/storage.yaml` with backend defaulting to local
- [ ] On re-run (idempotent): overwrites `.briefcase/` contents only; never modifies consumer `src/`, `tests/`, `docs/`, or any file outside `.briefcase/` and `./agent`; does not duplicate `.gitignore` entries
- [ ] `storage.yaml` lives at `.briefcase/storage.yaml` (not project root); all CLI commands resolve config from this path
- [ ] `./agent` entry point at project root invokes the CLI correctly regardless of Python environment and uses the same interpreter resolution contract as the generated wrapper
- [ ] If `NOTION_API_KEY` or legacy `NOTION_API_TOKEN` already exists in environment, install detects that and documents `NOTION_API_KEY` as the expected variable name for new setup
- [ ] Non-interactive by default: all required values can be passed as flags; prompts only when a required value is missing and no flag was provided
- [ ] Install prints a clear summary: what was created, what was skipped, and what the user needs to do manually
- [ ] Depends on `briefcase-namespace-isolation` being implemented first or in the same delivery slice
## Out of Scope
Provisioning the Notion workspace (that is agent setup); creating `storage.yaml` with live Notion credentials (user must set token manually); migrating existing installs from old layout to `.briefcase/` (that is the upgrade command); Windows support in the first version
## Open Questions
All resolved. `install.sh` is the canonical v1 installer, `.briefcase/` is treated as the installed framework namespace and runtime/config sentinel, `./agent` is generated by the installer, and v1 relies on the consumer's existing Python environment rather than creating a dedicated `.venv` automatically.
## Technical Approach
Use `install.sh` as the primary v1 entry point. The script should run without requiring a pre-installed Python package and should work both as a checked-in script and as a streamed shell installer. It should copy the framework source folders into `.briefcase/` as a post-install artifact rather than restructuring the framework repo itself.
Install should generate `./agent` using the entry-point contract from the companion brief: bash wrapper, project-root discovery via `.briefcase/`, interpreter preference for `.briefcase/.venv/bin/python` when present, and fallback to `python3`. The generated wrapper should bootstrap `src.cli.main` from `.briefcase/src/` by putting `.briefcase/` on `sys.path`.
For config, install should create `.briefcase/storage.yaml` with `backend: local` as the default. It should not write secrets into the config file. If `NOTION_API_KEY` already exists, report that setup is ready for Notion-backed commands; if only `NOTION_API_TOKEN` exists, treat it as a legacy alias and tell the user that `NOTION_API_KEY` is the canonical name going forward.
`.gitignore` updates must be additive and idempotent. Append the specific framework-owned/generated entries with short comments explaining why each exists, and avoid overwriting the consumer's existing ignore rules. The installer may overwrite `.briefcase/` and regenerate `./agent` on re-run, but it must never mutate consumer-owned source trees or docs.
No new dependencies are required. Cost impact is zero beyond the existing optional Notion usage because the installer is local-only and does not provision cloud resources.
